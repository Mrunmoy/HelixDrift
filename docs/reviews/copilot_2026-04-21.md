**However, the case for doing Anchor v4 first is stronger than the brief acknowledges.** The reason is architectural lock-in: if the PC-side fusion pipeline is built now consuming `sync_us` timestamps that carry a −14 ms systematic bias and a 50 ms p99 jitter component, the fusion code will either (a) bake in a corrective constant that becomes a silent liability, or (b) be refactored later when anchor v4 lands. Half a day now buys correctness for the entire fusion and rendering stack that follows. The brief itself says *"the moment you actually deploy to capture tight motion"* (`RF_NEXT_STEPS_DESIGN_BRIEF.md` line 111) — but that moment is the **first app demo**, not some far-future milestone.

There is also a **tooling gap that is an immediate risk regardless of which direction is chosen**: the complete test harness (`capture_robust.py`, `rf_baseline.sh`, `rf_phase_d_multireset.sh`, `sync_error_analysis.py`) lives only in `/tmp/helix_tag_log/` (`RF_ROBUSTNESS_REPORT.md`, lines 254–264). This is one reboot or `/tmp` cleanup away from being lost. This is not in the four candidates but is the single highest-leverage 30-minute act before any pivot.

## 4. Assumptions in the Brief That May Mislead Prioritisation

**A. "−14 ms systematic bias is the driver of the 50 ms p99 span" (implicit in §2.4)**
The brief implies that fixing the bias will collapse the span from 50 ms to "sub-10 ms". This is plausible but incomplete. The reported bias range is −13.4 to −14.5 ms (`RF_ROBUSTNESS_REPORT.md` line 149) — a 1.1 ms spread across 9 tags. Since the bias is nearly uniform, it contributes *less than 2 ms* to the inter-tag span by itself. The remaining ~48 ms of p99 span comes from per-frame jitter (the fast/slow ACK path randomness). Anchor v4's midpoint estimator fixes both, but the brief's framing suggests only the bias is the problem. The fix is correctly identified; the mechanism is slightly mis-stated. This matters because it also means a software-only bias correction on the PC side would NOT fix the 50 ms p99.

**B. "50 ms p99 is workable for body mocap" vs. the requirements doc's < 1 ms target**
`docs/rf-sync-requirements.md` line 17 states *"< 1 ms inter-node skew for multi-node kinematic chains"* as the requirement. The brief declares 50 ms p99 "workable but not pretty" (line 113). These are 50× apart. The brief is written by someone who has seen the real hardware numbers; the requirements doc is a draft v0.1 from a month earlier with no validation. Neither document explicitly supersedes the other. For fast-motion capture (golfer, martial artist: ~400 deg/s peak angular velocity), 50 ms of inter-node jitter produces ~20° of joint-angle error at p99 — twice the product target of ±10° in `docs/pose-inference-recommendation.md` line 62. The "workable" claim is only safe for slow-motion use cases.

**C. "The system is rock-solid for what it's doing now" (brief §4, line 166)**
This is narrowly true for the RF transport layer but mischaracterises the overall system state. M3 (Node Runtime) is ~50% complete (`SPRINT8_MILESTONE_REDIRECT.md` line 17), M4 (RF/sync simulation) has not started, and there is no PC-side ingest, fusion, or rendering. The project is not "rock-solid" — it is "solid at one layer." Declaring readiness to pivot to app work is appropriate, but the "rock-solid" framing may create a false sense of end-to-end system maturity for anyone reading the brief without the sprint context.

**Q1: Is 19 ms median / 50 ms p99 cross-tag span acceptable for 50 Hz mocap?**

The brief does not commit to a use case, which is the core of the ambiguity. For the stated v1 use cases in `pose-inference-recommendation.md` (§2.1 — VR/AR, film, sports biomechanics) the answer is "acceptable for film/VR, NOT acceptable for sports."

**Q2: Is the −14 ms systematic bias worth eliminating for its own sake?**
No, not for its own sake — because the bias is nearly identical across all 9 tags (−13.4 to −14.5 ms), so it contributes < 2 ms to inter-tag span and cancels in relative joint-angle calculations. However, anchor v4 is still worth doing because (a) the midpoint estimator also eliminates the ~48 ms of fast/slow-path jitter that IS driving the 50 ms p99 span, and (b) starting the fusion layer with a broken single-sample estimator is a design smell that will require either a known-bias constant in the fusion code or a later refactor.

**Q3: Missing failure modes not yet tested?**
Yes, three significant ones:
- **Long-duration / thermal**: no capture longer than 10 minutes exists. The 4-hour battery life target in `pose-inference-recommendation.md` (line 64) has zero measurement support. Crystal drift accumulation over hours is also uncharacterised.
- **2.4 GHz coexistence**: no test with WiFi router or BT headphones in the same room. Mandatory before any public demo or user trial.
- **Battery sag on Tags**: as LiPo voltage drops from 4.2 V to 3.2 V, ESB TX power and MCU clock accuracy may change. No data. Tag 5's failure pattern (`RF_ROBUSTNESS_REPORT.md` line 100) references the charge IC (`datasheets/TP4054.pdf`) — this failure mode is already latent in the fleet.

**Q4: For Phase E, should Hub-address discovery use BLE instead of ESB rendezvous?**
No — not for v1. BLE discovery would require the Tag to be in BLE mode during ESB mocap startup, but `CLAUDE.md` line 26 explicitly states *"Radio is shared: ESB or BLE, never both at same time."* Re-architecting the startup sequence to do a BLE discovery pass before entering ESB mode adds substantial risk for what is already a solved problem via Option A (flash-provisioned, ~2 h). BLE discovery is an elegant v2 feature for re-pairing in the field without SWD access; it is not the right v1 scope.

**Q5: Cheapest RF instrumentation to leave behind before pivoting to app work?**
In priority order: (1) commit the `/tmp` test harness to `tools/` (30 min, zero code change, prevents catastrophic tooling loss); (2) add `sync_error_us` as a signed field in the Hub's USB CDC FRAME line (one firmware field, gives PC app real-time sync quality); (3) document the TIFS fast/slow-path hypothesis in `docs/RF_PROTOCOL_REFERENCE.md` with a clear link to the sync_error_analysis script; (4) add a per-tag `frames_since_last_rx` counter to the Hub's CDC health output so the app can alert on a dead/degraded tag without post-hoc analysis.

1. **What is the peak angular velocity of the primary v1 target activity?** If the use case is "slow yoga / VR avatar" (≤ 100 deg/s), 50 ms p99 is acceptable and Anchor v4 drops from #1 to "nice-to-have."
